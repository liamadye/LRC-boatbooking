# LRC Boat Booking Portal — v1.1 Suggestions

After a thorough review of the v1.0 application, the following improvements are recommended, organised by priority and category.

---

## 1. Booking Flow Simplification

### 1.1 Remove "Number in boat" field — auto-derive from boat type
**Priority: High | Effort: Low**

The booking modal currently asks users to manually enter "Number in boat / crew" (1–9). This is redundant — the maximum crew is already defined per boat type in `MAX_CREW` (e.g. a 1x is always 1, an 8+ is always 9). The crew count is only used for validation and the totals bar.

**Suggestion:** Replace the numeric input with the boat's max crew auto-populated. For multi-configuration boats (e.g. `4x/4-/4+` which could be 4 or 5), show a simple radio or dropdown of the valid crew sizes. For single-configuration boats, just display the crew size as read-only info ("Crew: 9 (8 + cox)") — no input needed.

**Files affected:**
- `app/src/components/booking-modal.tsx` — remove `crewCount` input, auto-set from boat type
- `app/src/lib/constants.ts` — potentially add friendly crew labels
- `app/src/lib/validation.ts` — simplify crew count validation (now just a sanity check)
- `app/src/app/api/bookings/route.ts` — default `crewCount` from boat type if not provided

## 2. Authentication & User Permissioning

### 2.1 Invite-only registration (replace open sign-up)
**Priority: High | Effort: Medium**

Currently anyone can sign up via email/password or Google OAuth and immediately gets a `member` role with `recreational` access. This is a private club application and should be invite-only.

**Suggestion:** Implement an invitation system:
- **Admin sends invite** — admin panel gets a new "Invite Member" form (email, role, member type, squad)
- **Invite record** — new `Invitation` model storing the invited email, token, expiry, and who invited them
- **Login page** — remove the "Sign up" toggle entirely; only show "Sign In" and Google OAuth
- **Registration via invite link** — invited users receive an email with a `/register?token=xxx` link that pre-fills their email and allows them to set a password or link Google
- **Auto-create profile from invite** — when the user registers via the invite link, their `User` record is created with the role/memberType/squad specified in the invite, rather than defaulting to `member`/`recreational`
- **Block uninvited sign-ups** — if someone signs up through Supabase without an invite token, they should be blocked or placed in a `pending` state that requires admin approval

### 2.2 Admin link visibility — hide from non-admin users
**Priority: High | Effort: Low**

The "Admin" nav link is currently shown to **all users**. The page itself redirects non-admins, but showing the link creates confusion.

**Suggestion:** Pass the user's role to `AppNav` and conditionally render the Admin link only for `admin`, `captain`, and `vice_captain` roles.

**Files affected:**
- `app/src/components/app-nav.tsx` — accept `userRole` prop, filter `navLinks`
- `app/src/(app)/layout.tsx` — pass role to `AppNav`

### 2.3 Role-based permissions matrix
**Priority: Medium | Effort: Medium**

Currently permission checks are scattered across individual API routes with ad-hoc `if` statements. As the app grows, this will become harder to maintain.

**Suggestion:** Create a centralised permissions module (`lib/permissions.ts`) that defines:
- Which roles can perform which actions (book, cancel any booking, manage boats, manage users, send invites)
- Squad-level permissions (e.g. squad captains can manage bookings within their squad)
- A `can(user, action, resource?)` helper used consistently across all API routes

## 3. Booking Validation & Data Integrity

### 3.1 PATCH endpoint should re-run validation
**Priority: High | Effort: Low**

The `PATCH /api/bookings/[id]` endpoint allows editing `crewCount`, `endSlot`, etc. but does **not** re-run the validation rules. A user could edit a booking to exceed crew limits or create slot conflicts.

**Suggestion:** Run `validateBooking()` on the updated values before saving. Also check for slot conflicts with other bookings when `endSlot` is changed.

**Files affected:**
- `app/src/app/api/bookings/[id]/route.ts`

### 3.2 Weight validation uses only booker's weight, not actual crew average
**Priority: Medium | Effort: High**

The weight check currently compares the booking user's own weight against the boat's `avgWeightKg`. For crew boats (2+, 4+, 8+), this should be the average of the actual crew.

**Suggestion (short-term):** Add a note to the UI that weight validation is based on the booker's weight only, as a rough guide. CAN WE PARK THE WEIGHT VALIDATION FOR NOW?

**Suggestion (long-term):** Allow the booker to list crew members (by selecting other registered users), and calculate the true average from their profiles.

### 3.3 Booking window enforcement
**Priority: Medium | Effort: Low**

The `BookingWeek` model has `opensAt` and `closesAt` fields, but these are not enforced in the booking API — any user can book at any time. The Pymble Notes banner displays to users but doesn't prevent early/late bookings.

**Suggestion:** Enforce `BookingWeek.opensAt` / `closesAt` in `POST /api/bookings`. Show a clear message: "Bookings for this week open on [date]."

## 4. UX Improvements

### 4.1 Mobile responsiveness
**Priority: High | Effort: Medium**

The booking grid is a wide HTML table designed for desktop. On mobile devices it requires extensive horizontal scrolling and is essentially unusable.

**Suggestion:** Add a mobile-optimised view:
- On narrow screens, switch from the grid to a **list view** grouped by time slot, showing only available boats
- Or allow users to select a boat first, then see its availability across the week
- The `WeekNav` day picker already works on mobile — build on that

### 4.2 Real-time or optimistic updates
**Priority: Low | Effort: Medium**

Currently, after creating a booking, the page does a full `router.refresh()`. If two users are booking simultaneously, one might not see the other's booking until refresh.

**Suggestion (short-term):** Add a visible "Last refreshed X minutes ago" indicator and a manual refresh button.

**Suggestion (long-term):** Use Supabase Realtime subscriptions to push booking changes to all connected clients.

### 4.3 Booking confirmation & details
**Priority: Medium | Effort: Low**

When a user books, they get a small toast notification. There's no email confirmation, no detailed receipt, and no way to view the full details of a booking from the grid (you have to go to My Bookings).

**Suggestion:**
- Clicking an existing booking cell should show a detail popover (booker name, crew count, notes, cancel button if owner)
- Optional: send email confirmation via Supabase or a transactional email service

### 4.4 Colour coding and visual hierarchy
**Priority: Low | Effort: Low**

The grid cells show booked slots as blue chips. It would help to visually distinguish:
- Your own bookings (e.g. blue) vs other people's bookings (e.g. grey)
- Race-specific bookings (e.g. small icon or border)
- Multi-slot bookings spanning across cells (currently repeated per cell)

## 5. Admin Enhancements

### 5.1 Booking management for admins
**Priority: Medium | Effort: Medium**

Admins can delete or edit any booking via the API, but there's no admin UI for this — they'd have to find the booking in the grid and use the same flow as regular users.

**Suggestion:** Add an admin "All Bookings" tab that shows a filterable table of all bookings (by date range, boat, user) with bulk actions (cancel, reassign).

### 5.2 Audit log
**Priority: Medium | Effort: Medium**

There's no record of who changed what. If an admin changes a user's role, grants Black boat eligibility, or cancels someone's booking, there's no trail.

**Suggestion:** Add an `AuditLog` model that records admin actions (who, what, when, old value, new value). Display in a read-only admin tab.

### 5.3 Squad management
**Priority: Low | Effort: Low**

Squads are seeded but there's no UI to manage them — add/rename/deactivate squads, or assign/remove users from squads. Currently this requires database access.

**Suggestion:** Add a Squads tab in the admin panel.

---

## 6. Security

### 6.1 Add Supabase Row-Level Security (RLS)
**Priority: High | Effort: Medium**

All security is currently application-layer only. If the Supabase connection string leaks or someone accesses the database directly, there are no guardrails.

**Suggestion:** Add RLS policies on key tables:
- `Booking`: users can only DELETE/UPDATE their own bookings (unless admin)
- `User`: users can only UPDATE their own profile fields
- Admin-only tables (`BlackBoatApplication` review, boat status changes) restricted to admin roles

### 6.2 Rate limiting on API routes
**Priority: Medium | Effort: Low**

There's no rate limiting on booking creation or sign-up endpoints.

**Suggestion:** Add basic rate limiting (e.g. via Vercel Edge middleware or `upstash/ratelimit`) to prevent abuse.

---

## 7. Technical Debt

### 7.1 Add tests
**Priority: High | Effort: Medium**

There are currently no tests. The validation logic in particular is critical and should be covered.

**Suggestion:** Add:
- Unit tests for `validation.ts` (all 10 rules)
- Integration tests for key API routes (create booking, cancel booking, admin actions)
- E2E tests for the core booking flow (login, navigate grid, book, cancel)

### 7.2 GET bookings endpoint has no auth check
**Priority: Medium | Effort: Low**

`GET /api/bookings` has no authentication check — anyone can fetch booking data without being logged in. While the data isn't highly sensitive, it exposes member names and booking patterns.

**Suggestion:** Add the same auth check used by other endpoints.

### 7.3 Error handling on booking edit
**Priority: Low | Effort: Low**

The booking grid doesn't currently support editing an existing booking inline — users must cancel and rebook. Adding an "Edit Booking" modal (reusing `BookingModal`) would improve the experience.
